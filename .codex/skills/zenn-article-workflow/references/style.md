# Zenn Style Notes

この repo の既存記事から抽出した、Zenn 記事を書くときの基準。

## Core Tone

- 日本語で自然に読む。硬すぎる技術文書ではなく、実体験ベースで書く。
- 先に背景や「なぜこの話を書くのか」を置く。
- 断定しすぎず、確度が低い箇所はその旨を明示する。
- 最後に学び、注意点、次に同じことをする人への示唆を置く。

## Common Moves

- 導入で困りごとやきっかけを明かす
- 本文では「何を調べたか」「どこで判断したか」を書く
- 表で比較した方が早い箇所は表にする
- コードや設定例は、本文の主張を支える最小限に絞る

## `tech` Articles

- `## はじめに` で背景と問題意識を置く
- 先に結論に近い整理を見せ、その後で詳細に入る
- Linux と Windows、標準構成と例外構成のような比較は、見出しか表で整理する
- 運用上の注意点は、本文後半で明確に分離する

## `idea` Articles

- きっかけと判断の流れを重視する
- 「やりたかったこと」「なぜ採らなかったか」をセットで書く
- 結果が地味でも、判断理由が読者価値になるなら残す

## Wording Preferences

- 読者に直接役立つ言い回しを優先する
- 誇張や煽りを避ける
- 未確認情報には「可能性が高い」「印象でした」「明確な記載は見つからなかった」のような逃がしを使う

## Endings

- `## おわりに` または同等の締めで学びを回収する
- 読者が次に何を気をつければよいかで締める
